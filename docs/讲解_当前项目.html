① 一句话简介 + 技术栈

一套税务领域智能问答平台：用户一句自然语言进来，先做查询理解（QU），再由 确定性路由层（非 LLM）决定走哪条路——经营数据走 Text2SQL 查数仓、社保/商品编码走结构化查表（RAG 兜底）、 其余走文档 RAG 三级漏斗并逐 token 流式作答；同时通过 MCP 把核心能力工具化对外，并以默认关闭、物理隔离的方式接入联网补充。

核心框架 / 编排
FastAPI + SSE 双流（updates 进度 + messages 逐 token） LangGraph 主图 + Text2SQL 子图 确定性路由（IntentClassifier + SearchRouter，刻意不用 LLM supervisor） MCP（server 对外暴露 / client streamable-http 对内调）

★ 确定性路由层（非 LLM、不可被模型改写）：IntentClassifier（规则/正则优先，全不中才 LLM 兜底，异常回退默认意图） → SearchRouter 查静态映射表 SEARCH_INSTANCE_MAPPING（意图 → 查哪些库 / dense·sparse 权重 / 精排策略 / 是否挂结构化 Agent / 是否走 text2sql） → route_decider 一个 if：text2sql 走子图，否则走主图 RAG。
① 经营数据查询类 → Text2SQL 子图
route_type=text2sql，替换整条 RAG pipeline
1 schema_link（查 Qdrant 元数据）
2 generate_sql（LLM 生成）
3 ★ validate_fix（sqlglot AST 安全校验 / 修正，失败自环 ≤2 次）
4 execute（查 MySQL 数仓）
5 summarize（结论 + 伪流式）

产出 Text2SQLResult（sql/columns/rows/summary/error）；任何一步失败收敛进 error，不打断 SSE。

② 社保 / 商品编码 → 结构化查表 + RAG 兜底
route_type 仍是 rag，靠 RetrievalPlan.agent_name 挂查表 Agent
1 ★ 优先查表：ShebaoAgent / ProductCodeAgent 直查 MySQL
2 命中 → 行包成 Document，即作为召回结果（跳过文档召回）
3 无命中 → 回退文档 RAG 兜底
4 命中行 / 兜底文档统一进 粗排/精排/重排/摘要

查表 Agent 是"召回的一路输入"（数据源），非与 QA 平级的独立子智能体。当前实现是结构化优先：命中即用查表结果、跳过文档召回，仅无命中才回退文档 RAG。可选经 MCP 改道（失败回退直连 MySQL）。

④ 主-子 Agent 能力地图（确定性主 Agent + 多形态子能力）
★ 确定性 Supervisor 路由层（主"Agent"）：扮演别家"LLM supervisor"做的事——决定"这句话交给谁、怎么检索"， 但刻意用确定性实现（规则表 + 静态映射 + 一个 if）。理由：可控/可解释/零额外 token、权重是经验沉淀、安全边界清晰（联网开关不能由 LLM 决定）。

🧮 Text2SQLAgent 替换式子图

意图=经营数据时替换整条 RAG，独立 LangGraph 子图翻 SQL 查数仓。

🏦 ShebaoAgent 召回源

社保基数/比例直查 MySQL，命中行即作召回结果（结构化优先·跳过文档召回；非平级）。
接口：search(qu,topk)→list[Document]

🏷️ ProductCodeAgent 召回源

商品税收分类编码直查 MySQL，命中行即作召回结果（同构 Document，结构化优先）。
接口：search(qu,topk)→list[Document]

🌐 WebSearchAgent 隔离旁路

默认关·物理隔离，经 MCP 调联网搜索，纯加法兜底，绝不进权威引用。
接口：search(query)→list[Document]

base.py 用三个独立 Protocol（QaAgentLike / ReplacingSubgraphAgent / RecallSourceAgent）如实刻画三类接口——返回类型、调用语义、生命周期都不同，故不强行统一成单一 BaseAgent。

底层存储栈：Milvus（稠密向量）· ES（BM25 稀疏）· Qdrant（Text2SQL schema linking 元数据）· MySQL（数仓星型表 + 社保/编码表）· Mongo（多轮会话记忆）· MinIO（对象存储）。前 4 个 KBase 走文档召回，后 2 个走结构化查表，web 为隔离旁路。

⑦ ★ 端到端真跑验证状态（引用 e2e 报告逐例结论）
对真实运行的系统（conda tax_qa + Docker 全栈）跑 10 个例子，覆盖全部链路分支，9/10 在管线内跑通并给出正确答案； 第 10 条（联网兜底）在管线内未触发（本地召回恒充足，达不到"≤2"阈值），但联网搜索能力已单独验证可用（百炼 WebSearch MCP 返回真实结果）。 Text2SQL 数字与 MySQL 聚合一致；社保/编码查表命中真实表行；RAG 三级漏斗 + 本地 reranker + 逐 token 流式 + citation 全部真实运行。

诚实边界：答案"正确性"为人工核对演示语料（自洽假数据）所得；未做规模化评测（recall@k / 准确率 / QPS 等指标，本项目一律不写）。联网兜底在当前语料下不触发（属设计内保守门控，非缺陷，能力已单独验证可用）；Reranker 首次需联网下载模型（~600MB，已用 HF 镜像）。
