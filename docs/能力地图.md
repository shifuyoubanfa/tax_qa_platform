## 0. 一句话总览

这套系统【不是】"一个 LLM supervisor 调一堆工具"的多智能体编排，而是
**"确定性路由层 + 一张固定主图 + 几个形状各异的子能力"**。

```
                         ┌───────────────────────────────────────────────┐
   用户一句话  ─────────▶ │  确定性 Supervisor 路由层（非 LLM、不可被模型改写） │
                         │  IntentClassifier → SearchRouter → route_decider │
                         └───────────────────────────────────────────────┘
                                  │（按意图，确定性二选一）
                ┌─────────────────┴──────────────────────┐
                ▼（route_type=rag）                        ▼（route_type=text2sql）
   ┌───────────────────────────────────┐        ┌───────────────────────────────┐
   │  主图本体问答 QA（QaAgent 门面）       │        │  替换式子图 Text2SQLAgent        │
   │  load_memory→understand→route→      │        │  schema_link→generate_sql→      │
   │  rag_recall→coarse_rank→fine_rank→  │        │  validate_fix↻→execute→summarize│
   │  rerank→answerability_check⟂→       │        └───────────────────────────────┘
   │  build_context→generate_answer→     │
   │  save_memory                        │
   │  （⟂ 门控判不可答→low_confidence_     │
   │     answer 诚实兜底，见 §2.1）        │
   │     ▲                               │
   │     │ rag_recall 处并入召回池          │
   │  ┌──┴───────────────────────────┐    │
   │  │ 召回源 Agent（非平级，是数据源） │    │
   │  │ ShebaoAgent / ProductCodeAgent│    │
   │  └──────────────────────────────┘    │
   └───────────────────────────────────┘

   〔旁路 · 默认关 · 物理隔离〕联网搜索：确定性触发的隔离补充来源，结果绝不进权威引用（详见 §5）。
```

## 1. 确定性 Supervisor 路由层（本系统最值钱的资产）

**组成**：`IntentClassifier`（`app/core/qu/intent.py`）→ `SearchRouter`
（`app/core/router/search_router.py`）→ `route_decider`（`app/graph/nodes.py`）。

**它扮演的角色就是别家用"LLM supervisor"做的事**——决定"这句话该交给谁回答、怎么检索"。
但本系统【刻意用确定性实现，而不是 LLM supervisor】，正名为 **确定性 Supervisor 路由层**。

| 子件 | 输入 → 输出 | 怎么决策（确定性体现在哪） |
|---|---|---|
| `IntentClassifier` | query → Intent（7 类之一） | **规则/正则优先**（文号、书名号、固定术语…），规则全不中才调一次 LLM 兜底；任何异常回退默认意图 |
| `SearchRouter` | QUResult → RetrievalPlan | 查一张静态映射表 `SEARCH_INSTANCE_MAPPING`（意图→查哪些库 / dense·sparse 权重 / 粗精排策略 / 是否挂结构化召回源 / 是否走 text2sql） |
| `route_decider` | route_type → `"rag"` / `"text2sql"` | 一个 `if`：`route_type==text2sql` 走子图，否则走主图 RAG |

**为什么用确定性而不是 LLM supervisor（设计立场）**：

1. **可控/可解释/零额外 token**：路由是高频、延迟敏感、且"错一次就跑错整条链路"的关键岔口。
   规则表谁命中走哪条、为什么命中，一目了然、可单测、可复现；LLM supervisor 则把这个关键决策
   交给一个不可复现、可能臆造工具名/参数的黑盒。
2. **权重是经验沉淀**：精确法规类 `dense 0.05 / sparse 0.95`、通用问答 `dense 0.95 / sparse 0.05`
   这类配置是生产经验，写死在映射表里比让模型每次"猜"更稳。
3. **安全边界清晰**：联网搜索这类敏感旁路"开/关、何时触发"必须是确定的，**不能由 LLM 决定**（见 §5）。

> 红线（修订）：这一层的**确定性立场**是系统基座——**不引入** langgraph-supervisor / bind_tools 工具循环 /
> LLM 自由编排。红线约束的是"**关键决策不得交给 LLM 黑盒**"，而非"主图永不新增节点"：主图允许新增
> **确定性**节点。例如 §2.1 的「可答性门控」就是 2026-06 在重排后新增的一道**阈值门控**节点，它仍是
> 确定性判定（不调 LLM 裁决）、可单测可复现，与本立场完全一致。

## 2. 主图本体问答 QA —— `QaAgent`（薄门面）

| 项 | 说明 |
|---|---|
| **它是什么** | QA【就是主图本身】，不是"被调用的子图"。节点链：`load_memory → understand → route → rag_recall → coarse_rank → fine_rank → rerank → answerability_check →〔可答〕build_context → generate_answer → save_memory`；`answerability_check` 判〔不可答〕则转 `low_confidence_answer → save_memory`（见 §2.1） |
| **真实实现** | `app/graph/pipeline.py::Orchestrator`（`build_pipeline_graph` 编译主图，`astream` 真正执行） |
| **对外门面** | `app/agents/qa_agent.py::QaAgent` —— **零逻辑**，`astream()` 原样委托 `Orchestrator.astream`，`run()` 把 answer_delta 拼成整段。门面只提供"对外名字"，**不重写 RAG、不抽取节点、不改拓扑** |
| **接口形状** | `astream(req) -> AsyncIterator[SSEMessage]`（真·逐 token 流） |
| **流式机制** | `graph.astream(stream_mode=["updates","messages"])`：`updates` 映射 intent/retrieval/references/sql 事件；`messages` 里**仅** `generate_answer` 节点的 LLM token 逐个产出 `answer_delta` |

> 为什么单独给它一个 `QaAgent` 名字？仅为在能力地图上与 Text2SQLAgent / 召回源 Agent 并列登记、
> 便于对外描述。**名字在门面，实现仍在 Orchestrator**，两者解耦。

### 2.1 可答性门控 `answerability_check`（重排后/生成前的确定性"能不能答"判定）

| 项 | 说明 |
|---|---|
| **它是什么** | RAG 链路里**重排之后、生成之前**的一道**确定性阈值门控**（2026-06 新增）。漏斗只挑出"相对最相关"的 5 条，但不保证它们**真的相关**；门控据此决定"够不够答"——避免把弱证据喂给 LLM 产生带文号/数字却无依据的幻觉 |
| **真实实现** | `app/graph/nodes.py::answerability_check`（判定）+ `answerability_decider`（条件边）+ `low_confidence_answer`（不可答兜底）；阈值在 `config/settings.py`（`answerability_enabled` 默认开 / `answerability_min_score` 默认 0.15 / `answerability_min_docs`） |
| **怎么判（确定性）** | 主信号 = 重排 top1 相关性分（由 `RerankerClient` 经 sigmoid **已归一到 [0,1]**）。`≥ 阈值`→判可答、走原 `build_context→generate_answer`；`< 阈值 / 空召回`→判不可答、转 `low_confidence_answer` 给**诚实兜底话术**（不调 LLM、零编造） |
| **健壮性（坑与解）** | 重排客户端用 fp16+sigmoid，极不相关样本会**下溢成恰好 0.0**，本会与"重排降级未打分"的 0.0 难以区分 → `ReRanker` 成功打分钳到 `_SCORED_FLOOR`(1e-6>0)，使"成功打分"恒 > 0、"未打分"恒 = 0.0，**彻底解耦**（0.0 独占表示降级）；降级时改用"候选数"兜底判定，**宁放行不误杀**；NaN/inf 收敛为 0.0 |
| **为什么确定性而非 LLM** | 与 §1 立场一致：高频、延迟敏感、错一次就放幻觉的关键岔口，用**可单测/可复现/零额外 token** 的阈值判定，胜过又慢又不可复现的 LLM 黑盒 |
| **可观测** | 以 `answerability` SSE 事件透出 `answerable/confidence/signals`；前端对"判不可答"打"依据不足·诚实兜底"角标（只呈现门控自身字段，**不编造指标**） |
| **对标** | 对标企业财税客服的「问答库可答判断（阈值+大模型+规则）」环节，取其"确定性阈值"内核、刻意不引入 LLM 裁决 |

## 3. 替换式子图 Agent —— `Text2SQLAgent`（真子图）

| 项 | 说明 |
|---|---|
| **它是什么** | 一条**独立的回答路径**。当意图=`经营数据查询类(DATA_QUERY)` 时，由 `route_decider` 判定 `text2sql`，它【替换】整条 RAG pipeline，自己用一张内部 LangGraph 子图把自然语言翻成 SQL 查数仓 |
| **真实实现** | `app/agents/text2sql_agent.py::Text2SQLAgent` |
| **内部子图** | `schema_link → generate_sql → validate_fix（校验失败自环最多 2 次）→ execute → summarize`（条件边：通过→execute / 放弃→summarize / 需修→复验） |
| **接口形状** | `async arun(query, qu) -> Text2SQLResult`（结构化结果：sql/columns/rows/row_count/summary/error） |
| **与 QA 的关系** | **互斥二选一**（同一次请求要么走 QA 要么走它），不是 QA 的子步骤；它是"真子图"，所以叫替换式 |
| **健壮性** | 所有外部依赖惰性 + try/except；任何一步失败都收敛进 `Text2SQLResult.error`，不打断上层 SSE |

## 4. 召回源 Agent —— `ShebaoAgent` / `ProductCodeAgent`（**非平级子 Agent，是数据源**）

| 项 | 说明 |
|---|---|
| **它是什么** | 召回层的一个**结构化数据源**，不是与 QA 平级的"子智能体"。社保基数/比例、商品税收分类编码这类答案是 **MySQL 表里的精确值**，文档 RAG 给不出，所以挂一个查表 Agent |
| **真实实现** | `app/agents/structured_agents.py`（`StructuredAgent` 基类 + 两个子类 + `get_structured_agent` 工厂单例） |
| **怎么接入主图** | `SearchRouter` 在 RAG 计划里用 `RetrievalPlan.agent_name` 承载（route_type 仍是 **rag**）；`rag_recall` 节点读取它、**优先查表**，命中就把行包成 `Document` 并入召回池，一起进入 粗排/精排/重排/摘要；无命中才回退文档 RAG 兜底 |
| **接口形状** | `async search(qu, topk) -> list[Document]`（与文档召回同构；失败/无实体/无命中一律返回 `[]`，绝不抛异常） |
| **为什么"非平级"很重要** | 它的产物（Document）会与文档召回结果**合并后统一排序/摘要**，是"召回的一路输入"，而不是一个能独立回答用户的智能体。画成平级三兄弟会失真 |
| **可选 MCP 改道** | `settings.mcp_enabled=True` 时可经 MCP 工具查表（失败回退直连 MySQL），属优雅降级的可选层 |

## 5. 联网搜索：确定性触发的隔离旁路（**默认关 · 纯加法 · 已落地**）

**如实标注**：`app/agents/web_search_agent.py::WebSearchAgent` 已实现（经 MCP streamable http 调
联网搜索工具），并在主链路里以**纯加法兜底**方式接入。配套件：`config/settings.py` 的一组
**默认关闭**配置（`websearch_mcp_enabled=False` / `websearch_mcp_transport=http` /
`websearch_mcp_url` / `dashscope_api_key`）、`KBase.WEB` 来源标识、`McpClient` 的 http 传输分支。
它是一条**默认关、物理隔离**的补充来源旁路，不是与 QA/Text2SQL 平级的回答路径。

| 项 | 说明 |
|---|---|
| **真实实现** | `app/agents/web_search_agent.py::WebSearchAgent`，接口 `async search(query) -> list[Document]`（未开启/失败/超时一律返回 `[]`） |
| **传输层** | `app/clients/mcp_client.py` 新增 `http`（streamable http）分支，`list_tools/call_tool` 接受可选 `url/transport/api_key` 覆盖参数（向后兼容旧签名） |
| **何时触发** | `app/graph/nodes.py::rag_recall` 内 `_maybe_append_web_recall`：**三者同时满足**才追加——①`websearch_mcp_enabled=True` ②本地召回 `<= 2` 条 ③意图 ∈ {通用问题类, 政策汇集类}（确定性白名单） |
| **物理隔离** | 联网结果包成 `kbase=web` / `metadata.unverified=True` 的 Document；`summarizer.build_context` 先剥离 web 文档单独成"联网补充"区块，**不分配 `[[citation:N]]`、不进权威引用池**，权威编号与取材逻辑完全不变 |

它遵守的**硬契约**（红线，不可由 LLM 决定）：

1. **默认关**：总开关 `websearch_mcp_enabled=False`，关时 `WebSearchAgent.search` 直接返回 `[]`、**绝不触网**；`_maybe_append_web_recall` 整段跳过，**默认行为与未引入联网时完全一致**。
2. **物理隔离**：联网结果与本地权威法规库**物理隔离**，**绝不进权威法规引用 `[[citation:N]]`**（summarizer 剥离 + 独立区块标注）。
3. **明确标注**：补充来源统一标注 **"未核验 · 以官方为准"**（`_UNVERIFIED_SOURCE` / `_WEB_BLOCK_HEADER`）。
4. **确定性触发**：是否启用/何时触发由**确定性意图白名单**决定，**不由 LLM 决定**（与 §1 的安全边界一致）。
5. **纯加法 + 失败降级**：作为旁路新增，本地召回够多/不在白名单/超时/异常一律降级跳过，绝不破坏现有可跑链路。

## 6. 接口契约速查（对照 `app/agents/base.py`）

`base.py` 用三个独立的 `typing.Protocol` **如实刻画三类不同接口，不强行统一**（硬塞一个 BaseAgent
只会逼出"假统一"）。它只做类型/文档对照，**不要求现有类显式继承、不参与运行**。

| 能力 | Protocol（base.py） | 接口签名 | 返回 | 调用语义 |
|---|---|---|---|---|
| 主图本体问答 QA | `QaAgentLike` | `astream(req)` | `AsyncIterator[SSEMessage]`（逐 token 流） | 主图本体 |
| 替换式子图 | `ReplacingSubgraphAgent` | `arun(query, qu)` | `Text2SQLResult` | 替换整条 pipeline |
| 召回源 | `RecallSourceAgent` | `search(qu, topk)` | `list[Document]` | 并入召回池（非平级） |

三者**返回类型不同、调用语义不同、生命周期不同**——这正是"不统一成单一 BaseAgent"的原因。
